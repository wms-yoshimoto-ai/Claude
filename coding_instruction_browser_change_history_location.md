# 指示書: ブラウザ経由の変更履歴 地域詳細取得

## 背景

### API側の現状と制限

`fetch_change_history.py` で Google Ads API の `change_event` リソースから変更履歴を取得している。

**改修済み（2026-03-12）:**
- `change_event.old_resource` / `change_event.new_resource` を GAQL に追加
- CAMPAIGN_CRITERION の `criterionId` = `geo_target_constant` ID として地域名を自動解決
- 予算・tCPA・ステータス変更の旧値/新値も「変更詳細」列に出力

**APIの制限:**
- **change_event は直近30日間のデータのみ保持**（これが最大の制約）
- 管理画面では最大2年分の変更履歴を表示可能
- 30日以前の地域変更詳細を取得するにはブラウザ経由が必要

### 既存ブラウザデータの課題

既にブラウザ経由で取得した072の変更履歴データ（`072_pmax_change_history_20251217_20260209.json`）が存在するが:
- 「4 件の除外地域ターゲティングが追加されました」のようにサマリー表示のみ
- **具体的な地域名（例: 豊見城市、八重瀬町）が取得できていない**
- detail レコードでも件数のみで地域名は不明

---

## 目的

管理画面の変更履歴レポートから、**地域変更の具体的な地域名**をブラウザ経由で取得する。
特に30日以前の変更について、API では取得できない地域詳細を補完する。

---

## 取得フロー

### Step 1: 管理画面の変更履歴を開く

URL: `https://ads.google.com/aw/history?ocid={customer_id_without_hyphens}`

フィルタ設定:
- 期間: 必要な期間を指定（30日以前のデータが対象）
- 変更元: 全て
- 変更の種類: 「キャンペーン ターゲット」で絞り込むと効率的

### Step 2: 地域変更のレコードを特定

変更履歴一覧で以下のキーワードを含むレコードを探す:
- 「ターゲット」
- 「地域」
- 「近隣地域」

### Step 3: 変更の詳細を展開して地域名を取得

各変更レコードの「詳細を表示」や展開アイコンをクリックし、具体的な地域名を取得する。

**取得する情報:**
1. 日時
2. ユーザー（メールアドレス）
3. キャンペーン名
4. 操作（追加/削除）
5. 地域名（例: 豊見城市、八重瀬町）
6. 除外/対象の区分
7. 可能であれば geo_target_constant ID（DOMの属性やURLから取得）

### Step 4: 管理画面の変更詳細の表示パターン

管理画面では変更詳細は通常以下のように表示される:

```
変更前:            変更後:
（なし）           豊見城市
                   八重瀬町
                   南風原町
                   西原町
```

または削除の場合:
```
変更前:            変更後:
嘉手納町           （なし）
沖縄市
北谷町
```

これらのテキストを正確に取得すること。

## API版データとのマージ

API側の `fetch_change_history.py` の出力JSONには以下の構造で地域情報が含まれる:

```json
{
  "rows": [
    {
      "日時": "2026/02/14 06:46:47",
      "キャンペーン": "072 矯正 all（沖縄）",
      "変更内容": "キャンペーンのターゲットを作成しました",
      "変更詳細": "新: 八重瀬町, 沖縄県, 日本(ID:9190268)（除外）",
      "_resource_type": "CAMPAIGN_CRITERION",
      "_operation": "CREATE"
    }
  ]
}
```

マージ時のルール:
- 同一日時・同一キャンペーン・同一操作のレコードがAPI版にあればスキップ
- API版にないレコード（30日以前）はブラウザ版から追加
- `source` フィールドで出所を区別（"api" / "browser"）

## 技術的注意事項

1. **geo_target_constant ID**: DOMの属性（data-criterion-id等）やリンクURLに含まれる場合がある。取得できれば保存する。
2. **ページネーション**: 変更履歴が多い場合、スクロールやページ送りが必要。
3. **Pmax vs 検索**: 管理画面では両方の変更履歴が表示される。キャンペーン名で区別する。
4. **近隣地域ターゲット**: 「沖縄県那覇市安里２丁目６−２７から半径15km」のような形式。これも取得対象。
5. **複数アカウント対応**: site_id をパラメータ化し、072以外でも使えるようにする。

## 補足: criterion_id と geo_target_constant の関係

API調査（2026-03-12確認）で判明した重要な事実:

- `change_event` の `new_resource.campaignCriterion` は `criterionId` のみ返し、`location.geoTargetConstant` は含まない
- しかし、CAMPAIGN_CRITERION の LOCATION 型では **`criterionId` = `geo_target_constant` ID**
- つまり criterion_id をそのまま `geoTargetConstants/{id}` として名前解決可能
- この仕様を `fetch_change_history.py` に反映済み（"地域ID=" として出力 → resolve_geo_names() で名前解決）

ブラウザ取得時に geo_target_constant ID を取得できた場合、この対応関係を利用してAPI版データとの照合が可能。
